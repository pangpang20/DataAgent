## 记录 1：用户认证与权限管理需求文档编写

### 修改目标

编写 DataAgent 系统的用户认证与权限管理模块需求文档（Requirement_02），为后续安全体系开发提供完整的需求规格说明。

### 修改原因

当前系统不存在任何用户认证与权限管理机制——无 Spring Security、无用户表、无 JWT、无 RBAC。所有 `/api/*` 接口完全开放，`userId` 由客户端传入且不做验证，存在严重安全风险。需要一份企业级需求文档作为研发实施依据。

### 涉及文件

| 文件 | 操作 | 说明 |
|---|---|---|
| `docs/requirements/requirement_02.md` | 新增 | 用户认证与权限管理系统需求说明（1034 行） |

### 技术方案

文档涵盖以下设计：

- **认证方案**：基于 JWT（Access Token + Refresh Token）的无状态认证
- **权限模型**：标准 RBAC（用户 → 角色 → 权限 → 菜单），支持多对多关联
- **预置角色**：超级管理员、租户管理员、审核员、运维人员、普通用户、访客共 6 种
- **数据库设计**：10 张新表（sys_user、sys_role、sys_menu、sys_permission、4 张关联表、2 张日志表）
- **多库适配**：分别提供 MySQL、PostgreSQL、达梦的建表 SQL，含类型差异对照
- **安全策略**：BCrypt 密码加密、登录锁定、Token 黑名单、防水平/垂直越权、操作审计
- **接口设计**：RESTful 风格，6 组接口（认证、用户、角色、菜单、Agent 授权），统一响应结构与错误码
- **改造影响**：识别 10 个需改造的现有文件，新增 Spring Security + JWT + Redis 依赖

### 风险影响

| 风险项 | 等级 | 说明 |
|---|---|---|
| 现有接口改造 | 高 | 所有 Controller 需添加权限注解，可能影响现有功能 |
| 前端改造量 | 中 | 动态菜单、权限指令、登录页面均需前端配合 |
| 数据库迁移 | 低 | 新增 10 张表，不影响现有表结构 |
| Redis 依赖引入 | 低 | Token 黑名单需要 Redis，本地开发可用 embedded-redis |

## 记录 7：达梦迁移脚本 IDENTITY_INSERT 修复

### 修改目标

修复达梦 RBAC 初始化数据脚本中 `sys_menu` 表插入显式 ID 时报错 `-2723`。

### 修改原因

达梦数据库对自增列（IDENTITY）赋值时必须显式开启 `SET IDENTITY_INSERT table_name ON`，否则报错 `-2723: 仅当指定列列表，且SET IDENTITY_INSERT为ON时，才能对自增列赋值`。`sys_menu` 的 INSERT 使用了显式 ID（1, 2, 3, 101 等）用于菜单层级引用，缺少此开关。

### 涉及文件

| 文件 | 操作 |
|---|---|
| `sql/migration/dameng/V3_rbac_init_data.sql` | 添加 `SET IDENTITY_INSERT sys_menu ON/OFF` |

### 风险影响

无风险，仅影响达梦部署。

## 记录 10：权限标识对齐（Phase 10）

### 修改目标

修复代码 `@PreAuthorize` 中使用的权限标识与数据库 `sys_permission.permission_key` 之间的不一致，确保 RBAC 权限控制真正生效。

### 修改原因

Spring Security 通过精确匹配 `GrantedAuthority` 字符串判断权限。代码中使用 `agent:update` 但数据库定义为 `agent:edit`，导致所有非超级管理员用户的权限校验失败。此外还有 7 个权限标识（`agent:query`、`agent:datasource`、`knowledge:manage` 等）在代码中使用但数据库中缺失。

### 涉及文件

### 技术方案

- **权限重命名**：`agent:edit` → `agent:update`（菜单按钮 + sys_permission）
- **新增权限**：`agent:query`、`agent:datasource`、`knowledge:manage`、`file:upload`、`semantic:model`、`prompt:config`、`system:log`
- **SUPER_ADMIN 自动关联**：通过 `SELECT r.id, p.id FROM sys_role r, sys_permission p WHERE r.role_key = 'SUPER_ADMIN'` 自动获得所有新权限
- **DatasourceController 细化**：查询→`datasource:list`，创建→`datasource:create`，修改→`datasource:edit`，删除→`datasource:delete`，测试→`datasource:test`

### 风险影响

| 风险项 | 等级 | 说明 |
|---|---|---|
| 已有部署兼容 | 中 | 已部署环境需重新执行初始化数据脚本，否则新增权限不生效 |
| 前端无影响 | 低 | 前端菜单权限与后端独立，不依赖 permission_key 命名 |
