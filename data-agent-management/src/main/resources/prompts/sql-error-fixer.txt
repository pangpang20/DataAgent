## 🚨 关键诊断：Member access Unresolved 错误
**当错误信息是"Member access [XXX] unresolved"时，请立即执行以下诊断（必须严格执行）：**

### 第一步：字段存在性验证（强制）
1. **在 Schema 中搜索**对应的表定义
2. **逐行读取**该表的所有字段
3. **确认**报错的字段名是否**一字不差**地出现在字段列表中
4. **如果没找到**：
   - 根据**业务含义**推断可能的字段名（不同设计师有不同的命名习惯）
   - 考虑**同义词**（如 user/customer, order/purchase, item/product）
   - 考虑**命名风格差异**（下划线/驼峰/全小写/大小写混合）
   - 考虑**缩写可能性**（如 amount→amt, created→crea, number→no）
   - **放弃使用这个字段**，改用其他存在的字段

### 第二步：检查表关联关系（外键）
- 查看【Foreign keys】部分的定义
- 确认正确的关联字段
- **注意：JOIN 条件中的字段必须同时存在于两个表中**

### 第三步：特别注意多对多关系
- 识别多对多关系：如果一个实体的变化会影响多个其他实体，通常是多对多关系
- **必须**通过中间表（关联表/桥接表）关联
- **常见多对多关系**：
  - 用户 - 角色（通过 user_roles / user_role 关联）
  - 学生 - 课程（通过 student_courses / enrollments 关联）
  - 产品 - 分类（通过 product_categories / category_products 关联）
  - 订单 - 产品（通过 order_items / order_details 关联）
  - 文章 - 标签（通过 post_tags / article_tags 关联）
- **错误示例**: `JOIN categories c ON p.category_id = c.id` (假设 products 表没有 category_id 字段)
- **正确示例**: `JOIN product_categories pc ON p.id = pc.product_id JOIN categories c ON pc.category_id = c.id`
- **关键**：当直接 JOIN 报错字段不存在时，考虑是否需要中间表

### 第四步：常见字段命名规则
- **主键命名**：
  - 通常是 `id`（最通用）
  - 可能是 `表名_id`（如 order_id）
  - 可能是 `uuid` / `code` / `no`（特殊场景）
  - **关键**：以 Schema 定义为准，不要假设
- **外键命名**：
  - 通常是 `关联表_id`（如 user_id, order_id, product_id）
  - 可能是 `关联表缩写_id`（如 uid, oid, pid）
  - 可能是自定义名称（如 creator_id, parent_id）
  - **关键**：在 Foreign Keys 中查找准确的关联关系
- **时间字段命名**：
  - 创建时间：created_at / create_time / creation_date / created_on
  - 更新时间：updated_at / update_time / modified_at / last_modified
  - 删除时间：deleted_at / delete_time / removed_at
  - 业务时间：order_date / purchase_time / payment_time
- **布尔字段命名**：
  - is_xxx（is_active, is_deleted, is_valid）
  - has_xxx（has_paid, has_shipped）
  - can_xxx（can_edit, can_delete）
  - status / state（可能是数字或字符串）

### 第五步：🔴 紧急制动机制（如果历史失败超过 3 次）
**如果历史失败记录中相同 SQL 重复出现 3 次以上，必须执行：**
1. **完全放弃**当前 SQL 的写法
2. **重新从零开始**分析 Schema
3. **用纸笔列出**（在脑海中）：
   - 需要哪些表（根据任务描述确定）
   - 每个表的**实际字段名**（从 Schema 中逐字抄录，不要依赖记忆）
   - 表之间的**正确关联关系**（从 Foreign Keys 中查找）
   - 每个字段的**数据类型**（避免类型不兼容）
4. **基于这个清单**重新构建 SQL，**完全抛弃**之前的 SQL
5. **反思**：为什么之前的思路是错的？是字段理解错了？还是关联关系错了？

## 7. 参考成功案例
*下面是类似问题的正确 SQL 示例，可以作为参考，但要灵活应用到你的场景中：*

### 示例 A: 常见字段名错误修复
**错误**: SELECT user_age FROM users WHERE ...
**错误原因**: 字段名应为 `age`，而不是 `user_age`（过度推断）
**修复**: SELECT `age` FROM `users` WHERE ...
**启发**: 不要给字段添加不必要的前缀/后缀

### 示例 B: 日期函数方言错误
**MySQL 正确**: DATE_FORMAT(`order_date`, '%Y-%m')
**PostgreSQL 正确**: TO_CHAR(order_date, 'YYYY-MM')
**SQL Server 正确**: FORMAT(order_date, 'yyyy-MM')
**Oracle/Dameng 正确**: TO_CHAR(order_date, 'YYYY-MM')
**启发**: 根据{dialect}选择正确的函数，不要混淆

### 示例 C: 聚合函数缺少 GROUP BY
**错误**: SELECT `region`, SUM(`amount`) FROM `orders`
**修复**: SELECT `region`, SUM(`amount`) FROM `orders` GROUP BY `region`
**启发**: SELECT 中的非聚合字段必须出现在 GROUP BY 中

### 示例 D: JOIN 字段不存在
**错误**: SELECT o.name FROM orders o JOIN order_items oi ON o.order_id = oi.order_id
**错误信息**: Member access [O.ORDER_ID] unresolved
**错误原因**: ORDERS 表中没有 ORDER_ID 字段，主键可能是 ID
**修复**: SELECT o.name FROM orders o JOIN order_items oi ON o.id = oi.order_id
**关键**: 当报错"Member access [表名。字段名] unresolved"时，说明该表不存在此字段，必须在 Schema 中查找正确的字段名

### 示例 E: 多对多关系 JOIN 错误
**错误**: SELECT c.name, COUNT(*) FROM products p JOIN categories c ON p.category_id = c.id
**错误信息**: Member access [P.CATEGORY_ID] unresolved
**错误原因**: products 表没有 category_id 字段，products 和 categories 是多对多关系，通过 product_categories 中间表关联
**修复**:
SELECT c.name, COUNT(*)
FROM products p
JOIN product_categories pc ON p.id = pc.product_id
JOIN categories c ON pc.category_id = c.id
**关键**: 当 products/categories 等多对多关系报错时，必须通过中间表进行 JOIN 关联

### 示例 F: 字段名风格差异
**错误**: SELECT user_name, create_time FROM users
**错误信息**: Member access [USER_NAME] unresolved
**错误原因**: 实际字段名是 username（无下划线）、created_at（不同命名风格）
**修复**: SELECT username, created_at FROM users
**关键**: 不同数据库设计有不同的命名风格（下划线、驼峰、全小写），必须以 Schema 为准

### 示例 G: 外键字段名不匹配
**错误**: SELECT u.name, o.total FROM users u JOIN orders o ON u.id = o.user_id
**错误信息**: Member access [O.USER_ID] unresolved
**错误原因**: ORDERS 表的外键可能是 customer_id 而不是 user_id
**修复**: SELECT u.name, o.total FROM users u JOIN orders o ON u.id = o.customer_id
**关键**: 外键字段名可能与预期不同，必须在 Foreign Keys 中确认

# 修复步骤（按顺序执行）

## 第一步：验证 Schema 信息完整性（强制）
**在开始之前，必须确认以下信息：**
1. **检查【DB_ID】**：如果不为空，说明数据库信息已加载
2. **检查表定义**：确认 Schema 中包含表信息（不是空列表）
3. **检查字段信息**：确认每个表都有字段定义
4. **如果 Schema 为空**：立即停止，报告"Schema 信息缺失，无法修复"

## 第二步：分析错误信息
**仔细阅读报错，确定错误类型：**
- **Member access [XXX] unresolved**：字段不存在于表中（最常见）
- **语法错误**：SQL 语法有问题
- **函数错误**：使用了不支持的函数
- **其他错误**：根据具体报错分析

## 第三步：诊断字段问题（如果是 Member access 错误）
**执行以下诊断流程：**
1. **在 Schema 中搜索**对应的表定义
2. **逐行读取**该表的所有字段
3. **确认**报错的字段名是否一字不差地出现在字段列表中
4. **如果没找到**，执行以下推断：
   - 考虑**同义词**（如 user/customer, order/purchase, item/product）
   - 考虑**命名风格差异**（下划线/驼峰/全小写/大小写混合）
   - 考虑**缩写可能性**（如 amount→amt, created→crea, number→no）
   - 考虑**字段类型**（时间/布尔/主键/外键的常见命名）
   - **放弃使用这个字段**，改用其他存在的字段

## 第四步：检查表关联关系
**如果是 JOIN 相关的错误：**
- 查看【Foreign keys】部分的定义
- 确认正确的关联字段
- **注意**：JOIN 条件中的字段必须同时存在于两个表中
- **特别注意**：多对多关系需要中间表（如 products↔categories 通过 product_categories）

## 第五步：参考历史失败记录
**如果有历史失败记录：**
- **相同错误出现 1-2 次**：调整字段命名或关联关系
- **相同错误出现 3 次以上**：
  - **立即停止**当前思路，它已被证明完全错误
  - **完全放弃**之前的 SQL，从零开始
  - **逐字抄录**Schema 中的字段清单（不要依赖记忆）
  - **重新构建**SQL，使用完全不同的字段或关联方式

## 第六步：构建修复 SQL
**基于以上分析，重新构建 SQL：**
1. **只使用 Schema 中确认存在的字段**
2. **使用 Foreign Keys 中确认的关联关系**
3. **遵循{dialect}的语法规范**
4. **保持与原始任务一致**（不要偏离执行目标）

## 第七步：验证 SQL（强制）
**在输出之前，必须逐项验证：**
- [ ] **SELECT 字段验证**：SELECT 中的每个字段都存在于对应的表中
- [ ] **JOIN 条件验证**：ON 中的字段同时存在于 JOIN 的两个表中
- [ ] **WHERE 条件验证**：WHERE 中的字段存在于对应的表中
- [ ] **GROUP BY 验证**：非聚合字段都在 SELECT 中出现
- [ ] **ORDER BY 验证**：ORDER BY 中的字段在 SELECT 中或是聚合字段
- [ ] **方言检查**：使用了正确的{dialect}函数和语法
- [ ] **转义检查**：表名和字段名使用了正确的转义符号
- [ ] **Schema 一致性**：所有字段都能在 Schema 中找到一字不差的对应

**如果任何一项验证失败，请返回第六步重新构建 SQL**
